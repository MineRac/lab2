# Система управления запасами

Фронтенд: Vite + React + TypeScript.  
Бэкенд: Express + TypeScript + JWT + Supabase/PostgreSQL.

## Что добавлено

- REST API на `http://localhost:4000/api`
- Авторизация через `/api/auth/login`
- JWT-токен в `localStorage`
- Защищённые endpoint'ы для склада, заказов, настроек и дашборда
- CRUD для складских товаров
- Создание и изменение статусов автозаказов
- Подключение backend к Supabase через `@supabase/supabase-js`
- SQL-схема и seed-данные: `server/supabase-schema.sql`
- Подключение фронта к API для входа, склада и автозаказов

## 1. Создать базу в Supabase

1. Открой Supabase Dashboard.
2. Создай новый проект.
3. Открой `SQL Editor`.
4. Скопируй весь файл `server/supabase-schema.sql`.
5. Выполни SQL.

Скрипт создаст таблицы:

- `app_users`
- `inventory`
- `orders`
- `settings`

И добавит тестовые данные, включая пользователя:

- Логин: `admin`
- Пароль: `admin`

## 2. Настроить `.env`

Создай `.env` из примера:

```bash
cp .env.example .env
```

Заполни Supabase-переменные:

```env
API_PORT=4000
CLIENT_ORIGIN=http://localhost:5173
JWT_SECRET=change-this-secret-in-production
VITE_API_URL=http://localhost:4000/api

SUPABASE_URL=https://YOUR_PROJECT_ID.supabase.co
SUPABASE_SERVICE_ROLE_KEY=YOUR_SERVICE_ROLE_KEY
SUPABASE_ANON_KEY=YOUR_ANON_KEY
```

Где взять значения:

- `SUPABASE_URL`: Supabase Dashboard → Project Settings → API → Project URL
- `SUPABASE_SERVICE_ROLE_KEY`: Supabase Dashboard → Project Settings → API → `service_role` key
- `SUPABASE_ANON_KEY`: Supabase Dashboard → Project Settings → API → `anon` key

`SUPABASE_SERVICE_ROLE_KEY` должен оставаться только на backend. Не добавляй его в переменные с префиксом `VITE_`, иначе ключ попадёт во фронтенд-бандл.

## 3. Быстрый запуск

```bash
npm install
npm run dev:all
```

После запуска в dev-режиме:

- Frontend: `http://localhost:5173` — открывать приложение здесь
- Backend API: `http://localhost:4000/api`

- Healthcheck: `http://localhost:4000/api/health`

Демо-доступ:

- Логин: `admin`
- Пароль: `admin`

Если нужно, чтобы frontend и backend открывались с одного адреса `http://localhost:4000`, собери frontend и запусти сервер:


Команда сначала выполнит `vite build`, затем Express будет отдавать собранный frontend из папки `dist`.

## Важное для production

Текущая версия подключена к Supabase как рабочий прототип. Для production стоит заменить plaintext-пароли на `bcrypt`/`argon2`, добавить полноценные роли и права доступа, включить RLS-политики под выбранную модель доступа, добавить миграции, логирование, rate limiting и тесты.


## Поставщики

У поставщиков есть поле `delivery_days` — срок доставки в днях. При создании ручного заказа дата доставки рассчитывается по выбранному поставщику.


## Планирование закупок

Во вкладке «Планирование закупок» система показывает товары, у которых текущий остаток ниже минимального запаса. Для каждого товара можно выбрать поставщика, изменить рекомендуемое количество и создать заказ со статусом `scheduled`.


### Логика исключения из планирования

Товар не показывается во вкладке «Планирование закупок», если по нему уже есть заказ со статусом `pending`, `processing`, `scheduled` или `paused`.


## Цена ручного заказа

При создании ручного заказа цена подтягивается из выбранного товара в `inventory`, а сумма заказа рассчитывается как `price * quantity` и сохраняется в `orders.order_value`.


## Удаление отменённых заказов

Заказ можно удалить только после отмены. Backend проверяет статус и разрешает `DELETE /api/orders/:id` только для заказов со статусом `cancelled`.


## Мобильный список заказов

На экранах меньше `md` таблица заказов заменяется карточками. Десктопная таблица остаётся для больших экранов.


## Завершение заказов

Заказы со статусом `processing` можно перевести в `completed` кнопкой «Выполнить». Backend сохраняет изменение через `PATCH /api/orders/:id/status`.


## Пополнение склада при выполнении заказа

При переводе заказа в статус `completed` backend ищет товар в `inventory` по названию заказа и увеличивает `quantity` на количество из заказа. После изменения остатка пересчитывается статус товара: `В наличии`, `Низкий запас` или `Критический`.


## Статусы объёма на дашборде

В блоке «Топ-5 товаров по объёму» бейджи показывают уровень объёма на складе: `Критический`, `Средний`, `Высокий`. Это отдельная визуальная классификация для дашборда, не заменяющая основной складской статус товара.


## Категории товаров

В разделе «Складские запасы» можно открыть меню «Добавить категорию», создать категории и удалять их. В форме добавления товара поле «Категория» теперь является выпадающим списком и заполняется из таблицы `categories`.


## Excel-friendly CSV

Экспорт склада формирует CSV с UTF-8 BOM, строкой `sep=;` и переносами `CRLF`, чтобы Excel корректно открывал файл по колонкам в русской/европейской локали.


## CSV для Excel в Windows

Экспорт склада теперь формирует CSV в кодировке Windows-1251 с разделителем `;`, чтобы русская версия Excel корректно открывала кириллицу при двойном клике по файлу.

## ML-автозаказы и подтверждение

Endpoint `POST /api/ml/auto-orders/run` проверяет товары ниже минимального запаса и создаёт ML-заказы.

Логика статуса:
- `approval_required = true` → заказ создаётся в статусе `pending`
- `approval_required = false` → заказ создаётся в статусе `processing`

Учитываются настройки:
- `ml_enabled`
- `auto_order_enabled`
- `ml_confidence_threshold`
- `min_order_quantity`
- `max_order_quantity`
- `order_buffer`

Повторный заказ по товару не создаётся, если по этому товару уже есть активный заказ со статусом `pending`, `processing`, `scheduled` или `paused`.

## Мини-интерфейс симуляции продаж

Добавлена новая вкладка `Симуляция продаж`.

Что делает:
- позволяет выбрать товар;
- указать количество, скидку и дату продажи;
- записывает продажу в `sales_history`;
- уменьшает остаток товара в `inventory`;
- пересчитывает статус товара;
- создаёт движение склада в `stock_movements`;
- показывает последние продажи.

Backend endpoint:

```txt
POST /api/sales-history/simulate
```

Тело запроса:

```json
{
  "productId": "SKU001",
  "quantity": 2,
  "discountPercent": 5,
  "soldAt": "2026-06-08T12:00:00.000Z"
}
```


## Seed товаров для sales_history

Если `sales_history` заполнена, но в аналитике или симуляции продаж не видно нормальных данных, проверь таблицу `inventory`.

Для учебной базы продаж добавлен seed:

```txt
server/seed/inventory_products_for_sales_history.sql
```

Он создаёт товары `SKU001` ... `SKU015`, которые совпадают с `product_id` в `sales_history_2y`.

Рекомендуемый порядок загрузки:

1. `server/supabase-schema.sql`
2. `server/seed/inventory_products_for_sales_history.sql`
3. `server/seed/sales_history_2y_split/00_clear_sales_history.sql`
4. `server/seed/sales_history_2y_split/01_sales_history_insert.sql` ... до последнего файла

Также аналитика теперь умеет строить распределение категорий по `sales_history`, если `inventory` пустой.


## Период аналитики от последней продажи

Исправлена логика страницы «Аналитика»: финансовые KPI и график за 6 месяцев теперь строятся не от текущей даты компьютера, а от последней даты продажи в `sales_history`.

## Единая логика ML-метрик

Исправлена путаница между разными процентами:

- `Средняя уверенность` в AI Автозаказах = средний `aiConfidence` по AI-заказам.
- `Точность системы` в AI Автозаказах = доля успешных AI-заказов (`completed` или `processing`) среди всех AI-заказов.
- `Точность ML` на панели управления = та же успешность AI-заказов.
- В ML Прогнозах карточка `Точность модели` переименована в `Уверенность прогнозов`, потому что она показывает среднюю confidence из `ml_forecasts`, а не точность автозаказов.

Теперь показатели не конфликтуют между вкладками.
